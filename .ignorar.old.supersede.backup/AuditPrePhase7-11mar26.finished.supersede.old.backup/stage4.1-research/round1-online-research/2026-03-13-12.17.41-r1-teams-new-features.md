## 1. Agent Teams

### 1.1 TeamCreate API

**Enable:** `CLAUDE_CODE_EXPERIMENTAL_AGENT_TEAMS=1` in env or `settings.json`:
```json
{ "env": { "CLAUDE_CODE_EXPERIMENTAL_AGENT_TEAMS": "1" } }
```

**Requires:** Claude Code v2.1.32+. Released ~February 5, 2026.

**TeamCreate parameters:**
- `team_name` — namespace identifier; links tasks, messages, and config
- `description` — purpose of the team
- `agent_type` — type of agents to spawn

**Storage created:**
- `~/.claude/teams/{team-name}/config.json` — members array (name, agent ID, agent type)
- `~/.claude/tasks/{team-name}/` — task list directory

---

### 1.2 SendMessage Tool

**Message types:**

| Type | Purpose | Key Params |
|------|---------|-----------|
| `message` | Direct DM to one teammate | `recipient`, `content`, `summary` |
| `broadcast` | All teammates simultaneously | `content`, `summary` |
| `shutdown_request` | Ask teammate to shut down gracefully | `recipient`, `content` |
| `shutdown_response` | Approve/reject shutdown | `request_id`, `approve`, optional `content` |
| `plan_approval_response` | Approve/reject teammate's plan | `request_id`, `recipient`, `approve`, optional `content` |

**Notes:**
- Messages auto-delivered; recipients don't poll
- Broadcast cost scales linearly with team size — use sparingly
- Plain text output is NOT visible to teammates; must use SendMessage tool
- Idle notifications sent automatically when a teammate stops

---

### 1.3 Architecture Constraints

- **No nested teams:** teammates CANNOT spawn their own teams or additional teammates
- **One team per session:** a lead can only manage one team at a time
- **Lead is fixed:** the session that creates the team is lead for its lifetime; no promotion/transfer
- **Permissions set at spawn:** all teammates start with lead's permission mode; can change individually after spawn but not at spawn time
- **Context isolation:** each teammate has its own context window; does NOT inherit lead's conversation history
- **In-process teammates cannot resume:** `/resume` and `/rewind` do not restore in-process teammates
- **Task status can lag:** teammates sometimes fail to mark tasks completed — monitor and nudge
- **Split panes NOT supported in:** VS Code integrated terminal, Windows Terminal, Ghostty

---

### 1.4 Roles (Lead / Orchestrator / Worker)

| Role | Description |
|------|-------------|
| **Team Lead** | Main Claude Code session; creates team, spawns teammates, coordinates work, runs cleanup |
| **Teammate (worker)** | Full independent Claude Code session; claims tasks, executes work, messages peers |
| **No distinct Orchestrator role** | Lead IS the orchestrator — no sub-orchestrator concept in native teams |

**Spawning context:** teammates load same project context as a regular session (CLAUDE.md, MCP servers, skills) but receive only the spawn prompt from lead — not conversation history.

---

### 1.5 Delegate Mode

**Shift+Tab** toggles the lead into coordination-only mode — prevents lead from implementing tasks itself, forcing pure delegation to teammates.

**In-process navigation:**
- `Shift+Down` — cycle through teammates (wraps back to lead after last)
- `Enter` — view a teammate's session
- `Escape` — interrupt teammate's current turn
- `Ctrl+T` — toggle task list

**Split-pane mode:** click into any pane to interact directly. Requires tmux or iTerm2 + `it2` CLI.

**Display mode setting:**
```json
{ "teammateMode": "in-process" }   // or "tmux" / "auto" (default)
```

---

### 1.6 Task Tools

**Task states:** `pending` → `in_progress` → `completed` (plus dependency blocking)

| Tool | Purpose | Key Params |
|------|---------|-----------|
| `TaskCreate` | Define a work unit | `subject`, `description` (prompt teammates read), dependencies |
| `TaskUpdate` | Claim, complete, or delete a task | `status`, `owner` (agent name); v2.1.x added delete capability |
| `TaskGet` | Get details of a specific task | task ID |
| `TaskList` | List all tasks with status/ownership | — |
| `TaskOutput` | Read output from a background task | task ID |
| `TaskStop` | Stop a running task | task ID |

**File locking:** used to prevent race conditions when multiple teammates try to claim the same task simultaneously.

**Dependency tracking:** when a task completes, dependent blocked tasks automatically unblock.

---

### 1.7 Team Hooks

| Hook | Trigger | Exit Code 2 Behavior |
|------|---------|---------------------|
| `TeammateIdle` | When a teammate is about to go idle | Sends feedback to teammate and keeps it working |
| `TaskCompleted` | When a task is being marked as completed | Prevents completion and sends feedback |

**TeammateIdle** and **TaskCompleted** support `{"continue": false, "stopReason": "..."}` to stop teammates.

These hooks enable quality gates: e.g., auto-run tests when a teammate finishes a module, block task completion if coverage is below threshold.

---

### 1.8 Anti-Patterns

| Anti-Pattern | Why Bad | Fix |
|-------------|---------|-----|
| Multiple teammates editing the same file | Leads to overwrites and conflicts | Each teammate owns separate file sets |
| Unattended long runs | Increases risk of wasted effort, diverging approaches | Monitor and steer; check in frequently |
| Too many teammates (>5-6) | Coordination overhead > parallel benefit; token cost scales linearly | Start with 3-5; scale only if genuinely needed |
| Lead implementing instead of delegating | Defeats purpose of team | Use Shift+Tab delegate mode; instruct lead to wait |
| Task granularity too small | Coordination overhead exceeds benefit | Size tasks as self-contained deliverables (function, test file, review) |
| Broadcasting instead of DMing | Expensive: N teammates = N deliveries | Default to `message` type; broadcast only for critical team-wide alerts |
| Teammates spawning sub-agents | Not supported; will fail | Lead handles all spawning |

### 2.4 Agent Changes

- **Worktree isolation:** `isolation: "worktree"` in agent definitions — auto-creates and cleans up isolated git worktrees
- **Background execution:** `background: true` always runs agent as background task
- **Memory scope:** `memory: "user" | "project" | "local"` — controls persistence scope
- **Tool restriction:** `Task(AgentName)` syntax to disable specific agents from tool list
- **Per-agent hooks:** agents can define `PreToolUse`, `PostToolUse`, `Stop` hooks in frontmatter
- **Team model inheritance:** teammates inherit the leader's model by default
- **Full model IDs:** `claude-opus-4-5`, `claude-sonnet-4-6` etc. now accepted in `model:` field
- **Skills in agents:** skills support `context: fork` and `agent` field for execution
- **Teammates cannot spawn agents:** architectural constraint — no nested teams

---

### 2.5 Permission Changes

- **Wildcard Bash patterns:** `Bash(npm *)`, `Bash(* install)`, `Bash(git * main)` now work
- **`Bash(*)` equivalent to `Bash`** — accepted as full permission
- **Managed policy rules** no longer bypassable by user `allow` rules
- **Interactive tools** in `allowed-tools` no longer auto-allowed without user prompts
- **`sandbox.excludedCommands`** and `dangerouslyDisableSandbox` now blocked from `autoAllowBashIfSandboxed`
- **Unreachable rules detection** — `/doctor` warns about redundant/unreachable permission rules
- **`allowManagedDomainsOnly`** blocks non-allowed domains automatically with no bypass
- **`permissions.disableBypassPermissionsMode`** — new setting to prevent bypass mode entirely
- **Better error messages** showing actual restriction reasons

|------|---------|----------------|
| `/output-style` command | v2.1.73 | Use `/config` instead |
| npm installations | v2.1.15 | Use `claude install` (native install) |
| Opus 4 and 4.1 models | v2.1.68 | Auto-migrated to Opus 4.6 |
| Bash login shell `-l` flag | v2.1.51 | Skipped by default when shell snapshot available |
| Tool results persistence threshold | v2.1.51 | Reduced from 100K → 50K characters |
| Managed settings path (Windows) | — | `C:\ProgramData\ClaudeCode\` → `C:\Program Files\ClaudeCode\` |
| Slash commands / skills | v2.1.3 | Merged; unified with no behavioral change |
