# Round 1 Research: Skills, CLAUDE.md & Workflow
> Source: Official Claude Code docs (code.claude.com/docs) — verified March 2026

## 1. Skills System

### 1.1 Skills 2.0 Format

Skills are defined in a `SKILL.md` file inside a named directory under `.claude/skills/`:

```
.claude/skills/<skill-name>/
├── SKILL.md           # Required entrypoint
├── template.md        # Optional supporting file
├── examples/
└── scripts/
    └── helper.sh      # Scripts Claude can execute
```

**Key change (v2.1.3 — Jan 2026):** Custom slash commands (`.claude/commands/deploy.md`) were **merged into the skills system**. Both formats still work and produce the same `/deploy` slash command. Skills take precedence if name conflicts. Skills add features commands lack: supporting files, invocation control, subagent execution, dynamic context injection.

Skills follow the [Agent Skills](https://agentskills.io) open standard, extended by Claude Code.

### 1.2 Frontmatter Fields

All fields are optional except `description` (strongly recommended):

| Field | Required | Description |
|-------|----------|-------------|
| `name` | No | Slash-command name (dir name used if omitted). Max 64 chars, lowercase + hyphens only. |
| `description` | **Recommended** | What skill does and when to use it. Used for auto-invocation decisions. |
| `argument-hint` | No | Autocomplete hint, e.g. `[issue-number]` |
| `disable-model-invocation` | No | `true` = user-only invocation (Claude cannot auto-invoke). Default: `false` |
| `user-invocable` | No | `false` = hidden from `/` menu, Claude-only. Default: `true` |
| `allowed-tools` | No | Tools allowed without per-use approval when skill is active |
| `model` | No | Override model for this skill |
| `context` | No | Set to `fork` to run in isolated subagent |
| `agent` | No | Which subagent type to use with `context: fork` (`Explore`, `Plan`, `general-purpose`, or custom) |
| `hooks` | No | Hooks scoped to this skill's lifecycle |

**Invocation control matrix:**

| Frontmatter | User invoke | Claude invoke | Context loading |
|-------------|-------------|---------------|-----------------|
| (default) | Yes | Yes | Description always in context |
| `disable-model-invocation: true` | Yes | No | Description NOT in context |
| `user-invocable: false` | No | Yes | Description always in context |

### 1.3 Invocation

- **Direct**: `/skill-name [args]` — user types the slash command
- **Auto**: Claude loads skill automatically when conversation matches `description`
- **Arguments**: `$ARGUMENTS` (all args), `$ARGUMENTS[N]` or `$N` (positional)
- **Dynamic context**: `!`command`` syntax executes shell commands before Claude sees content
- **Subagent fork**: `context: fork` runs skill in isolated context (no conversation history)

String substitutions available: `$ARGUMENTS`, `$ARGUMENTS[N]`, `$N`, `${CLAUDE_SESSION_ID}`, `${CLAUDE_SKILL_DIR}`

### 1.4 Discovery and Loading

**Storage locations (priority order: enterprise > personal > project > plugin):**

| Level | Path | Scope |
|-------|------|-------|
| Enterprise | Managed settings | All org users |
| Personal | `~/.claude/skills/<name>/SKILL.md` | All your projects |
| Project | `.claude/skills/<name>/SKILL.md` | This project only |
| Plugin | `<plugin>/skills/<name>/SKILL.md` | Where plugin enabled |

- Skill descriptions are loaded into context at session start (within a 2% context window budget, fallback 16,000 chars)
- Full skill content loads only when invoked
- Subagents with `skills` preloaded: full content injected at startup
- **Monorepo support**: skills in nested `.claude/skills/` dirs discovered automatically when editing files in that subdirectory
- `--add-dir` directories: skills auto-loaded, CLAUDE.md not loaded by default
- Live change detection: skills edited during session are picked up without restart
- Override budget: `SLASH_COMMAND_TOOL_CHAR_BUDGET` env var

## 2. CLAUDE.md

### 2.1 Purpose and Loading

CLAUDE.md files give Claude persistent instructions across sessions. Loaded at start of every session into the context window. Treated as context (advisory), not enforced config. More specific + concise = better adherence.

Two memory systems (both loaded at session start):
1. **CLAUDE.md files** — instructions you write
2. **Auto memory** (`~/.claude/projects/<project>/memory/MEMORY.md`) — notes Claude writes itself (first 200 lines loaded)

### 2.2 File Discovery Order

Claude walks the directory tree from working dir upward, loading all CLAUDE.md files found:

| Scope | Location | Notes |
|-------|----------|-------|
| Managed policy | `/Library/Application Support/ClaudeCode/CLAUDE.md` (macOS) | Cannot be excluded |
| Personal | `~/.claude/CLAUDE.md` | All projects, loaded at launch |
| Project | `./CLAUDE.md` or `./.claude/CLAUDE.md` | Team-shared via git |
| Ancestor dirs | `../../CLAUDE.md`, etc. | Loaded at launch (walking up) |
| Subdirectory | `./src/CLAUDE.md` | Loaded **on demand** when Claude reads files in that dir |

Rules in `.claude/rules/` are loaded at launch (or on demand if `paths` frontmatter scopes them).

### 2.3 @ References

`@path/to/file` syntax imports files inline. Expanded and loaded into context at launch alongside CLAUDE.md.

- Both relative and absolute paths allowed
- Relative paths resolve from the file containing the import (not working dir)
- Recursive imports supported — max depth: **5 hops**
- First encounter with external imports shows approval dialog; if declined, imports stay disabled
- Can import from home dir: `@~/.claude/my-project-instructions.md`

### 2.4 Global vs Project Level

| Level | File | What to put here |
|-------|------|-----------------|
| Global | `~/.claude/CLAUDE.md` | Personal preferences, personal tooling shortcuts |
| Project | `./CLAUDE.md` or `./.claude/CLAUDE.md` | Build/test commands, architecture, coding standards, workflows, team conventions |
| Rules | `.claude/rules/*.md` | Topic-specific files, can scope to file paths |

Personal preferences **should not** go in project CLAUDE.md (would affect teammates).

### 2.5 Recommended Structure

No required format. Keep it short and human-readable. Target: **under 200 lines**.

✅ Include:
- Bash commands Claude can't guess
- Code style rules that differ from defaults
- Testing instructions / preferred test runners
- Repository etiquette (branch naming, PR conventions)
- Architectural decisions specific to project
- Developer environment quirks (required env vars)
- Common gotchas / non-obvious behaviors

❌ Exclude:
- Anything Claude can infer from reading code
- Standard language conventions Claude already knows
- Detailed API docs (link instead)
- Information that changes frequently
- Long tutorials

Use emphasis (`IMPORTANT`, `YOU MUST`) for critical rules. Check CLAUDE.md into git.

### 2.6 Compact Instructions

Add a section like:
```
## Compact Instructions
When compacting, always preserve the full list of modified files and any test commands
```
After `/compact`, Claude re-reads CLAUDE.md fresh from disk — CLAUDE.md **fully survives compaction**. Only conversation-only instructions are lost.

### 2.7 CLAUDE.local.md

Not a separate file type — Claude Code uses `.claude/settings.local.json` for local-only exclusions. For personal overrides, import a file from home dir:
```markdown
# Individual Preferences
- @~/.claude/my-project-instructions.md
```
This keeps personal settings off version control while referencing them in the shared CLAUDE.md.

**Note**: `claudeMdExcludes` in `.claude/settings.local.json` can skip specific CLAUDE.md files (useful for monorepos with other teams' files). Managed policy CLAUDE.md cannot be excluded.

## 3. Workflow Files

### 3.1 Recommended Structure

The `.claude/` directory is the primary extension point. Official recommended layout:

```
.claude/
├── CLAUDE.md              # Main project instructions
├── settings.json          # Project settings
├── settings.local.json    # Local-only settings (gitignored)
├── skills/                # Custom skills (slash commands)
│   └── <skill-name>/
│       └── SKILL.md
├── agents/                # Custom subagent definitions
│   └── <agent-name>.md
├── rules/                 # Topic-specific instruction files
│   ├── code-style.md
│   ├── testing.md
│   └── security.md
└── worktrees/             # Git worktrees (gitignore this dir)
```

The `.claude/workflow/` directory used in this project is a **custom convention** (not an official Claude Code structure). Official docs do not prescribe a `workflow/` subdirectory.

### 3.2 Common Patterns

From official docs and community practice:

**Session-start pattern**: Not a built-in Claude Code concept. Implemented via:
- CLAUDE.md `## Workflow` section with session trigger instructions
- Skills with auto-invocation (description matches session-start scenarios)
- Hooks: `SessionStart` event fires shell commands automatically at every session

**Before-commit pattern**: Implemented via:
- Pre-commit hooks in `.claude/settings.json`
- Skills with `disable-model-invocation: true` (e.g. `/commit`, `/verify`)
- CLAUDE.md listing verification commands

**Human checkpoints pattern**: Implemented via:
- Skills or agents using `AskUserQuestion` tool
- Plan Mode (`--permission-mode plan`) for safe read-only exploration before acting
- CLAUDE.md instructions to pause at specific milestones

**Verification agents pattern**: Implemented via:
- Custom subagents in `.claude/agents/`
- Skills with `context: fork` to run in isolated contexts
- Agent teams for parallel execution

### 3.3 Reference Patterns

**Auto-loaded (every session):**
- CLAUDE.md and all ancestor CLAUDE.md files
- `.claude/rules/*.md` files without `paths` frontmatter
- First 200 lines of `~/.claude/projects/<project>/memory/MEMORY.md`

**On-demand loaded:**
- Subdirectory CLAUDE.md files (when Claude reads files in that dir)
- `.claude/rules/*.md` with `paths` frontmatter (when matching files opened)
- Skill full content (when skill invoked)
- Auto memory topic files (Claude reads when needed)

**Best practice for workflow files**: Reference them from CLAUDE.md using `@` imports or as skills. Avoid auto-loading large workflow files that only apply sometimes — use skills with `disable-model-invocation: true` for explicit-trigger workflows, or path-scoped rules for file-type-specific guidance.

**The `/init` command** generates a starter CLAUDE.md by analyzing the codebase. On existing CLAUDE.md, suggests improvements rather than overwriting.
