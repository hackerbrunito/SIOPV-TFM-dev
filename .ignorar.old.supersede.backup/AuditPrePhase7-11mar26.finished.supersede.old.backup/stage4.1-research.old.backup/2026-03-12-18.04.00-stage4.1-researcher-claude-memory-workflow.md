## TOPIC 1: CLAUDE.md Patterns

### Global vs Project-Level CLAUDE.md

Claude Code supports multiple CLAUDE.md scopes (precedence: managed policy > project > user):

| Scope | Location | Purpose | Shared with |
|---|---|---|---|
| **Managed policy** | macOS: `/Library/Application Support/ClaudeCode/CLAUDE.md`; Linux/WSL: `/etc/claude-code/CLAUDE.md`; Windows: `C:\Program Files\ClaudeCode\CLAUDE.md` | Organization-wide standards managed by IT/DevOps | All users in organization (cannot be excluded) |
| **Project instructions** | `./CLAUDE.md` or `./.claude/CLAUDE.md` | Team-shared instructions for the project | Team via source control |
| **User instructions** | `~/.claude/CLAUDE.md` | Personal preferences for all projects | Just you (all projects) |

- **Global (`~/.claude/CLAUDE.md`)**: Called "user memory". Applies to ALL projects. Contains global coding style, language preferences, personal tooling shortcuts, naming conventions. Does NOT contain project-specific architecture.
- **Project-level (`./CLAUDE.md` or `./.claude/CLAUDE.md`)**: Contains project architecture, build/test commands, coding standards, common workflows. Shared via git. Should focus on project-level standards, not personal preferences.
- **Additional subdirectory CLAUDE.md files**: Exist per-subdirectory in large projects. Loaded on-demand (not at launch) when Claude reads files in those subdirectories.

### @file Import Syntax

Exact syntax: `@path/to/file` — place the `@` prefix anywhere in the CLAUDE.md text.

```text
See @README for project overview and @package.json for available npm commands.

# Additional Instructions
- git workflow @docs/git-instructions.md
```

Behavior:
- Imported files are expanded and loaded into context at launch alongside the CLAUDE.md that references them.
- Both relative and absolute paths are allowed.
- Relative paths resolve relative to the file containing the import (NOT the working directory).
- Imported files can recursively import other files, with a **maximum depth of five hops**.
- The first time Claude Code encounters external imports, it shows an approval dialog listing the files. If declined, imports stay disabled and the dialog does not appear again.
- For personal preferences not committed to source control: import a file from home directory inside shared CLAUDE.md: `- @~/.claude/my-project-instructions.md`
- No other import/include mechanism besides `@file` is documented for CLAUDE.md files.

### Content Guidelines

**What belongs in CLAUDE.md:**
- Build and test commands
- Coding standards and style preferences
- Architectural decisions
- Naming conventions
- Common workflows
- Tech stack details
- Project-specific rules

**What does NOT belong:**
- Personal preferences in project-level files (put in `~/.claude/CLAUDE.md` instead)
- Content shared with a specific team in user-level files
- Very large instruction sets (split using `@imports` or `.claude/rules/` instead)

**Writing guidelines:**
- Target under 200 lines per CLAUDE.md file (longer files consume more context and reduce adherence)
- Use markdown headers and bullets to group related instructions (organized sections are easier to follow)
- Write concrete, verifiable instructions: "Use 2-space indentation" not "Format code properly"
- Avoid contradicting instructions across files

### Auto-loading Behavior

- CLAUDE.md files in the directory hierarchy **above** the working directory are **loaded in full at launch**.
- CLAUDE.md files in **subdirectories** load on demand when Claude reads files in those subdirectories (NOT at launch).
- Claude Code walks up the directory tree from the current working directory, loading CLAUDE.md at each level.
- Example: running Claude Code in `foo/bar/` loads both `foo/bar/CLAUDE.md` and `foo/CLAUDE.md`.
- The `/memory` command lists all CLAUDE.md and rules files loaded in the current session for debugging.
- `.claude/rules/*.md` files (without `paths` frontmatter) are loaded at launch with the same priority as `.claude/CLAUDE.md`.
- Rules WITH `paths` frontmatter only load when Claude works with matching files (conditional loading).
- CLAUDE.md fully survives `/compact` — re-read from disk and re-injected into context after compaction.
- `CLAUDE_CODE_ADDITIONAL_DIRECTORIES_CLAUDE_MD=1` env var can load CLAUDE.md from directories added via `--add-dir`.

### Minimal Working Example

```text
# Project: MyAPI

## Build & Test
- Build: `npm run build`
- Test: `npm test`
- Lint: `npm run lint`

## Code Style
@.claude/rules/code-style.md

## Architecture
See @docs/architecture.md for system overview.

## Rules
- All API handlers live in `src/api/handlers/`
- Use 2-space indentation
- Run `npm test` before committing

## Personal Preferences
@~/.claude/my-project-instructions.md
```

### .claude/rules/ Directory

For large projects: organize instructions into topic-specific `.md` files in `.claude/rules/`. Each file covers one topic (`testing.md`, `api-design.md`). All `.md` files are recursively discovered. Supports YAML frontmatter with `paths` field for conditional loading:

```markdown
---
paths:
  - "src/api/**/*.ts"
---
# API Rules
```

User-level rules also supported at `~/.claude/rules/` (loaded before project rules, so project rules have higher priority).

### Changes from Previous Format

The official docs describe a 4-scope system (managed policy + project + user + subdirectory on-demand). The `.claude/rules/` path-specific rules with YAML frontmatter are current. The `/init` command auto-generates CLAUDE.md from codebase analysis.

No explicit changelog entry found marking a format break. SFEIR Institute and other secondary sources confirm the multi-level hierarchy.

## TOPIC 2: Memory File Patterns

### Directory Structure

Auto memory is stored at: `~/.claude/projects/<project>/memory/`

The `<project>` component is derived from the git repository path. All worktrees and subdirectories within the same repo share one auto memory directory. Outside a git repo, the project root is used.

```
~/.claude/projects/<project>/memory/
├── MEMORY.md          # Concise index, loaded into every session (first 200 lines)
├── debugging.md       # Detailed notes on debugging patterns
├── api-conventions.md # API design decisions
└── ...                # Any other topic files Claude creates
```

This is auto memory — Claude writes these files itself. It is **NOT** a `.claude/memory/` directory inside the project repo.

Note: The current SIOPV project appears to have a custom convention at `/Users/bruno/.claude/projects/-Users-bruno-sec-llm-workbench/memory/` with individual memory files using a specific frontmatter schema (`name`, `description`, `type`). This is a USER-DEFINED convention layered on top of the standard auto-memory mechanism, not a Claude Code standard.

### MEMORY.md Index Format

Official standard:
- `MEMORY.md` acts as an **index** of the memory directory
- First **200 lines** are loaded at the start of every conversation
- Content beyond line 200 is NOT loaded at session start
- Claude keeps MEMORY.md concise by moving detailed notes into separate topic files
- No frontmatter schema defined in official docs — MEMORY.md is plain markdown acting as index

The SIOPV project uses MEMORY.md as a structured index with section headers and tables, with individual memory files using frontmatter (see below). This is a project-specific convention, not Claude Code standard.

### Individual Memory File Format

**Official Claude Code standard**: Topic files (e.g., `debugging.md`, `patterns.md`) are plain markdown. They are NOT loaded at startup — Claude reads them on demand using standard file tools when needed. No frontmatter schema is required or documented in official sources.

**SIOPV project convention** (found in system context of this session — user-defined, not Claude Code standard):
```markdown
---
name: {{memory name}}
description: {{one-line description}}
type: {{user, feedback, project, reference}}
---

{{memory content}}
```

### Valid Memory Types

**Official Claude Code**: No formally defined memory types in the auto-memory system. Files are plain markdown topic files created by Claude as needed.

**SIOPV project convention** (user-defined):
- `user` — user role, preferences, knowledge
- `feedback` — corrections and guidance from the user
- `project` — ongoing work, goals, bugs, incidents
- `reference` — pointers to external systems

These 4 types are NOT official Claude Code types — they are a project-level convention found in the CLAUDE.md instructions for this session.

### Loading Behavior

- **MEMORY.md**: First 200 lines loaded at start of every conversation (auto-injected)
- **Topic files** (e.g., `debugging.md`): NOT loaded at startup — Claude reads them on demand during a session
- Auto memory is on by default; disabled via `{ "autoMemoryEnabled": false }` in settings or `CLAUDE_CODE_DISABLE_AUTO_MEMORY=1` env var
- Machine-local: not shared across machines or cloud environments
- Subagents can maintain their own auto memory (see subagent configuration)
- The `/memory` command allows browsing and editing memory files from within a session

### Minimal Working Examples

**MEMORY.md (official pattern):**
```markdown
# Project Memory

## Build System
- Build: `npm run build` (takes ~45s, uses esbuild)
- Test: `npm test` — requires Redis running locally on port 6379

## Architecture Notes
Detailed patterns: @api-conventions.md
Debugging history: @debugging.md
```

**Topic file (official pattern — no frontmatter required):**
```markdown
# API Design Conventions

- All endpoints return `{ data, error, meta }` shape
- Pagination via cursor, not offset
- Error codes documented in src/errors/codes.ts
```

**Individual memory file with frontmatter (SIOPV/user-defined convention):**
```markdown
---
name: user_role
description: user is a data scientist focused on observability
type: user
---

User is a data scientist, currently focused on observability/logging.
```

## TOPIC 3: Workflow File Conventions

### .claude/workflow/ — Standard or Project-Specific?

**FINDING: `.claude/workflow/` is NOT a standard Claude Code directory.**

It is a project-specific convention. Official Claude Code documentation makes zero mention of `.claude/workflow/` as a recognized directory. It has no special auto-load behavior, no official schema, and is not indexed by Claude Code.

The SIOPV project uses `.claude/workflow/` as an organizational container for markdown reference files. These files are loaded into Claude's context only when explicitly referenced via `@` imports in CLAUDE.md (for auto-load) or when the user/Claude reads them on demand (for on-demand access). The directory itself is just a folder of plain markdown files.

### Auto-loaded vs On-demand Files

**Official Claude Code mechanism** for loading structured context files:

| Mechanism | When Loaded | How |
|-----------|------------|-----|
| CLAUDE.md files (project root or `.claude/CLAUDE.md`) | Session start (auto) | Directory tree walk at launch |
| `.claude/rules/*.md` (no `paths` frontmatter) | Session start (auto) | Same priority as `.claude/CLAUDE.md` |
| `.claude/rules/*.md` (with `paths` frontmatter) | On-demand | When Claude works with matching files |
| Subdirectory CLAUDE.md files | On-demand | When Claude reads files in that subdirectory |
| `@file` imports in CLAUDE.md | Session start (auto) | Expanded and loaded alongside CLAUDE.md at launch |
| `.claude/skills/<name>/SKILL.md` | Descriptions at start; full content on invocation | Skills framework |
| `.claude/agents/*.md` | On-demand | When spawned as subagents |

**SIOPV project convention** (project-specific, not Claude Code standard):
- Files in `.claude/workflow/` are either:
  1. Referenced via `@` imports in CLAUDE.md → loaded at session start
  2. Listed in CLAUDE.md as "on demand" → user or Claude must explicitly request/read them
- Example from SIOPV CLAUDE.md: `@.claude/workflow/01-session-start.md` (auto-loaded), while `02-reflexion-loop.md` is "read on demand"

### .claude/rules/ vs .claude/docs/ vs .claude/workflow/

| Directory | Official Status | Claude Code Behavior |
|-----------|----------------|----------------------|
| `.claude/rules/` | **OFFICIALLY DOCUMENTED** | Auto-loaded at session start (or conditionally with `paths` frontmatter). All `.md` files recursively discovered. |
| `.claude/docs/` | NOT documented | No special behavior. Claude Code ignores it unless explicitly referenced via `@` imports or direct file reads. |
| `.claude/workflow/` | NOT documented | No special behavior. Claude Code ignores it unless explicitly referenced via `@` imports or direct file reads. |

The only way to make `.claude/docs/` or `.claude/workflow/` files available to Claude is:
1. Reference them with `@path` imports in CLAUDE.md (loads at session start)
2. Have the user or Claude explicitly read them during a session

### Other Standard .claude/ Subdirectories

Complete list of officially documented `.claude/` subdirectories:

| Directory | Scope | Auto-loaded? | Purpose |
|-----------|-------|-------------|---------|
| `.claude/CLAUDE.md` | Project | Yes (at launch) | Alternative location for main project instructions |
| `.claude/rules/` | Project | Yes (at launch; conditional with `paths`) | Modular instruction files, supports YAML `paths` frontmatter |
| `.claude/skills/<name>/` | Project | Descriptions at start; content on-demand | Custom skill definitions (SKILL.md per skill) |
| `.claude/commands/` | Project | Yes (legacy, merged into skills) | Legacy skill location, still works |
| `.claude/agents/` | Project | On-demand | Custom subagent definitions (markdown with YAML frontmatter) |
| `.claude/settings.json` | Project | Yes (at launch) | Project-level settings, committed to git |
| `.claude/settings.local.json` | Local | Yes (at launch) | Local project overrides, gitignored |

User-level equivalents (under `~/.claude/`):
- `~/.claude/CLAUDE.md` — user-level instructions
- `~/.claude/rules/` — user-level rules (loaded before project rules)
- `~/.claude/skills/` — user-level skills

Org-level (managed policy):
- macOS: `/Library/Application Support/ClaudeCode/CLAUDE.md`

NOT officially documented (no special behavior):
- `.claude/workflow/` — project-specific convention only
- `.claude/docs/` — project-specific convention only
- `.claude/hooks/` — hooks are configured in `settings.json` under `hooks:` key, not a directory

**Note on hooks**: Hooks are configured in `settings.json` (not a directory). The `hooks` key in settings maps events to shell commands. Some users put hook scripts in `.claude/hooks/` by convention, but this directory has no special status in Claude Code.

### Minimal Working Example

Since `.claude/workflow/` is not a standard Claude Code pattern, the equivalent standard pattern uses `.claude/rules/` (for auto-loaded instructions) and `@` imports (for on-demand reference files):

**Standard Claude Code approach (using official mechanisms):**
```text
.claude/
├── CLAUDE.md                    # Main instructions (auto-loaded)
├── rules/
│   ├── session-start.md         # Auto-loaded at every session
│   ├── coding-standards.md      # Auto-loaded at every session
│   └── before-commit.md         # Auto-loaded at every session
└── skills/
    └── deep-reference/
        └── SKILL.md             # On-demand, invoked with /deep-reference
```

**SIOPV project approach (using project-specific .claude/workflow/ convention):**
```text
.claude/
├── CLAUDE.md                    # References workflow files via @
├── workflow/
│   ├── 01-session-start.md      # @-imported in CLAUDE.md → auto-loaded
│   ├── 02-reflexion-loop.md     # Listed as "on demand" in CLAUDE.md
│   ├── 03-human-checkpoints.md  # Listed as "on demand" in CLAUDE.md
│   └── ...
└── rules/
    └── agent-reports.md         # Auto-loaded via .claude/rules/ standard
```

Both approaches achieve the same effect. The SIOPV approach predates or parallels the `.claude/rules/` standard.
