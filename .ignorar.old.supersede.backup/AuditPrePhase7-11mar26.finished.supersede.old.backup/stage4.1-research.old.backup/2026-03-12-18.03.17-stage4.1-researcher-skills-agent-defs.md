## TOPIC 1: Skills 2.0 Format

### Verified Schema

All fields are optional except `description` is strongly recommended. `name` is also optional (defaults to directory name).

| Field | Required | Type | Valid Values / Notes |
|---|---|---|---|
| `name` | No | string | Lowercase letters, numbers, and hyphens only; max 64 chars. If omitted, uses directory name. Becomes the `/slash-command`. |
| `description` | Recommended | string | What the skill does and when to use it. Claude uses this to decide when to apply the skill. If omitted, uses first paragraph of markdown content. |
| `argument-hint` | No | string | Hint shown during autocomplete. Example: `[issue-number]` or `[filename] [format]`. |
| `disable-model-invocation` | No | boolean | `true` prevents Claude from automatically loading this skill. Use for `/commit`, `/deploy`, etc. Default: `false`. |
| `user-invocable` | No | boolean | `false` hides from the `/` menu. Use for background knowledge users shouldn't invoke directly. Default: `true`. |
| `allowed-tools` | No | string | Comma-separated list of tools Claude can use without asking permission when skill is active. Example: `Read, Grep, Glob`. |
| `model` | No | string | Model to use when this skill is active. |
| `context` | No | string | `fork` to run in a forked subagent context. Only valid value documented is `fork`. |
| `agent` | No | string | Which subagent type to use when `context: fork` is set. Options include `Explore`, `Plan`, `general-purpose`, or any custom subagent from `.claude/agents/`. |
| `hooks` | No | object | Hooks scoped to this skill's lifecycle. |
| `version` | No | string | Seen in Context7 examples (e.g., `1.0.0`) and plugin-dev SKILL.md files. Not in official frontmatter schema table. |

**String substitutions available in skill body:**

| Variable | Description |
|---|---|
| `$ARGUMENTS` | All arguments passed when invoking the skill. If not present in content, args appended as `ARGUMENTS: <value>`. |
| `$ARGUMENTS[N]` | Specific argument by 0-based index. |
| `$N` | Shorthand for `$ARGUMENTS[N]` (e.g., `$0` for first, `$1` for second). |
| `${CLAUDE_SESSION_ID}` | The current session ID. |
| `${CLAUDE_SKILL_DIR}` | Directory containing the skill's SKILL.md file. |

**Shell command injection:** `` !`command` `` syntax runs shell commands before skill content is sent to Claude. Output replaces the placeholder (preprocessing, not Claude execution).

### File Location & Directory Structure

| Scope | Path |
|---|---|
| Enterprise | See managed settings docs |
| Personal | `~/.claude/skills/<skill-name>/SKILL.md` |
| Project | `.claude/skills/<skill-name>/SKILL.md` |
| Plugin | `<plugin>/skills/<skill-name>/SKILL.md` |

Priority when same name conflicts: enterprise > personal > project. Plugin skills use `plugin-name:skill-name` namespace (no conflict possible).

Each skill is a **directory** with `SKILL.md` as the required entrypoint:
```
my-skill/
├── SKILL.md           # Main instructions (required)
├── template.md        # Optional supporting file
├── examples/
│   └── sample.md      # Optional examples
└── scripts/
    └── validate.sh    # Optional scripts
```

**Filename convention:** Directory named after the skill slug (e.g., `explain-code`). File inside is always `SKILL.md`.

**Automatic discovery from nested dirs:** Claude Code also discovers skills from nested `.claude/skills/` directories within subdirectories where files are being edited (monorepo support).

### Minimal Working Example

```yaml
---
name: explain-code
description: Explains code with visual diagrams and analogies. Use when explaining how code works, teaching about a codebase, or when the user asks "how does this work?"
---

When explaining code, always include:

1. **Start with an analogy**: Compare the code to something from everyday life
2. **Draw a diagram**: Use ASCII art to show the flow, structure, or relationships
3. **Walk through the code**: Explain step-by-step what happens
4. **Highlight a gotcha**: What's a common mistake or misconception?

Keep explanations conversational. For complex concepts, use multiple analogies.
```

**Example with invocation control and arguments:**
```yaml
---
name: deploy
description: Deploy the application to production
disable-model-invocation: true
---

Deploy $ARGUMENTS to production:
1. Run the test suite
2. Build the application
3. Push to the deployment target
4. Verify the deployment succeeded
```

**Example with forked subagent:**
```yaml
---
name: deep-research
description: Research a topic thoroughly
context: fork
agent: Explore
---

Research $ARGUMENTS thoroughly:
1. Find relevant files using Glob and Grep
2. Read and analyze the code
3. Summarize findings with specific file references
```

**Example with tool restriction:**
```yaml
---
name: safe-reader
description: Read files without making changes
allowed-tools: Read, Grep, Glob
---

[skill instructions here]
```

### Changes from Previous Format (Skills 2.0 vs. Old `.claude/commands/` Format)

**Official statement from docs:** "Custom commands have been merged into skills. A file at `.claude/commands/deploy.md` and a skill at `.claude/skills/deploy/SKILL.md` both create `/deploy` and work the same way. Your existing `.claude/commands/` files keep working. Skills add optional features: a directory for supporting files, frontmatter to control whether you or Claude invokes them, and the ability for Claude to load them automatically when relevant."

**What changed — Skills 2.0 additions over old `.claude/commands/` format:**
1. Skills now use a **directory** structure (not a flat `.md` file). The `SKILL.md` lives inside a named directory.
2. Skills support **supporting files** alongside `SKILL.md` (templates, examples, scripts).
3. New frontmatter fields: `context`, `agent`, `allowed-tools`, `hooks`, `argument-hint`.
4. `disable-model-invocation` and `user-invocable` fields (also available in old commands format per docs).
5. Skills follow the **Agent Skills open standard** at `agentskills.io`, which works across multiple AI tools.
6. Claude Code extensions beyond the standard: invocation control, subagent execution, dynamic context injection (`` !`cmd` `` syntax), argument substitutions.
7. **If a skill and a command share the same name, the skill takes precedence.**

**Old format still works:** Flat `.md` files in `.claude/commands/` with frontmatter. No directory structure required. No supporting files.

### What to Avoid / Obsolete Content

1. Do NOT use `.claude/commands/` for new skills — use `.claude/skills/<name>/SKILL.md` instead (though old commands files still work).
2. Do NOT put the skill content in a flat `.md` file if you need supporting files — use the directory structure.
3. Do NOT use `user-invocable: false` to prevent Claude from invoking a skill programmatically — it only hides from the `/` menu. Use `disable-model-invocation: true` for that instead.
4. Do NOT include built-in commands like `/compact` and `/init` through the Skill tool — they are not available via the Skill tool.
5. Keep `SKILL.md` under 500 lines — move detailed reference material to separate files.
6. Do NOT use `context: fork` for skills that only contain reference guidelines without an actionable task — the subagent receives guidelines but no actionable prompt and returns without meaningful output.
7. Do NOT rely on the `user-invocable` field to block programmatic skill access — only `disable-model-invocation: true` blocks Claude from triggering the skill.

### How Skills Are Invoked

- **User invocation:** Type `/skill-name` to invoke directly, with optional arguments: `/fix-issue 123`.
- **Claude auto-invocation:** Claude loads the skill automatically when relevant to conversation (based on `description`). Only works if `disable-model-invocation` is not `true`.
- **Arguments:** Passed after the skill name, accessible via `$ARGUMENTS`, `$ARGUMENTS[N]`, or `$N`.
- **Subagents with preloaded skills:** Full skill content is injected at startup (not just descriptions). Subagents do not inherit skills from the parent conversation — must list explicitly via `skills` field.

## TOPIC 2: Agent Definition File Format

### Verified Schema

Only `name` and `description` are required.

| Field | Required | Type | Valid Values / Notes |
|---|---|---|---|
| `name` | Yes | string | Unique identifier using lowercase letters and hyphens |
| `description` | Yes | string | When Claude should delegate to this subagent. Write clearly so Claude knows when to use it. Include "use proactively" for eager delegation. |
| `tools` | No | string or array | Allowlist of tools the subagent can use. Inherits all tools from main conversation if omitted. Comma-separated string (e.g., `Read, Grep, Glob`) or array (e.g., `["Read", "Grep"]`). Use `Agent(worker, researcher)` syntax to restrict which subagents can be spawned. |
| `disallowedTools` | No | string or array | Denylist — tools to deny, removed from inherited or specified list. |
| `model` | No | string | `sonnet`, `opus`, `haiku`, or `inherit`. Defaults to `inherit` (uses same model as main conversation). |
| `permissionMode` | No | string | `default`, `acceptEdits`, `dontAsk`, `bypassPermissions`, or `plan`. |
| `maxTurns` | No | integer | Maximum number of agentic turns before the subagent stops. |
| `skills` | No | array | List of skill names to preload into the subagent's context at startup. Full skill content is injected (not just made available for invocation). Subagents don't inherit skills from parent. |
| `mcpServers` | No | array or object | MCP servers available to this subagent. Each entry is either a server name (referencing already-configured server) or an inline definition. |
| `hooks` | No | object | Lifecycle hooks scoped to this subagent. Same format as settings.json hooks. |
| `memory` | No | string | Persistent memory scope: `user`, `project`, or `local`. Enables cross-session learning. |
| `background` | No | boolean | `true` to always run as a background task. Default: `false`. |
| `isolation` | No | string | `worktree` to run in a temporary git worktree (isolated copy). Automatically cleaned up if no changes made. |
| `color` | No (UI only?) | string | Background color for the subagent in the UI (e.g., `blue`). Set via `/agents` interactive UI. Seen in Context7 examples as frontmatter field. **CONFLICT: not in official frontmatter schema table.** |

**CLI-defined subagents (JSON format):** The `--agents` JSON flag accepts the same fields as file-based frontmatter, plus `prompt` (equivalent to the markdown body):
```json
{
  "code-reviewer": {
    "description": "...",
    "prompt": "You are a senior code reviewer...",
    "tools": ["Read", "Grep"],
    "model": "sonnet"
  }
}
```

### File Location & Directory Structure

| Scope | Path | Priority |
|---|---|---|
| `--agents` CLI flag | Session only (JSON, not saved to disk) | 1 (highest) |
| Project | `.claude/agents/<name>.md` | 2 |
| Personal (user) | `~/.claude/agents/<name>.md` | 3 |
| Plugin | `<plugin>/agents/<name>.md` | 4 (lowest) |

When multiple subagents share the same name, higher-priority location wins.

**Filename convention:** `<name>.md` (e.g., `code-reviewer.md`). The filename slug typically matches the `name` frontmatter field.

**Note:** Subagents are loaded at session start. If you create a subagent by manually adding a file, restart your session or use `/agents` to load it immediately.

### File Body Format

The body (everything after the closing `---` frontmatter marker) is the **system prompt** in Markdown. Subagents receive only this system prompt plus basic environment details (working directory, etc.) — NOT the full Claude Code system prompt.

```markdown
---
name: code-reviewer
description: Expert code reviewer. Use immediately after writing or modifying code.
tools: Read, Grep, Glob, Bash
model: inherit
---

You are a senior code reviewer ensuring high standards of code quality and security.

When invoked:
1. Run git diff to see recent changes
2. Focus on modified files
3. Begin review immediately

Review checklist:
- Code is clear and readable
- No duplicated code
- Proper error handling
- No exposed secrets or API keys
- Good test coverage

Provide feedback organized by priority:
- Critical issues (must fix)
- Warnings (should fix)
- Suggestions (consider improving)
```

### Minimal Working Example

```markdown
---
name: code-reviewer
description: Reviews code for quality and best practices
tools: Read, Glob, Grep
model: sonnet
---

You are a code reviewer. When invoked, analyze the code and provide

```

```markdown

  - api-conventions
  - error-handling-patterns

Implement API endpoints. Follow the conventions and patterns from the preloaded skills.
```

### Valid Values for `model` Field

| Value | Behavior |
|---|---|
| `sonnet` | Use Sonnet model (balances capability and speed) |
| `opus` | Use Opus model (most capable) |
| `haiku` | Use Haiku model (fast, low-latency, low-cost) |
| `inherit` | Use same model as main conversation (default if field omitted) |

**Note on `bypassPermissions` mode:** This mode in agent definitions only works when the parent Claude Code session was started with `--dangerously-skip-permissions`. Without that session-level flag, `bypassPermissions` has no effect.

### Valid Values for `tools` Field

Any of Claude Code's internal tools. Common values:
`Read`, `Write`, `Edit`, `Bash`, `Grep`, `Glob`, `Agent`, `Skill`, `WebFetch`, `WebSearch`, `NotebookEdit`, `Task`, `TodoRead`, `TodoWrite`, etc.

Special syntax:
- `Agent(worker, researcher)` — allowlist of which subagents this agent can spawn (only relevant for main-thread agents, not subagents themselves)
- `Agent` (no parentheses) — allow spawning any subagent without restrictions
- Omit `Agent` entirely — agent cannot spawn subagents

Both comma-separated string format and JSON array format appear to be supported:
- String: `tools: Read, Grep, Glob, Bash`
- Array: `tools: ["Read", "Grep", "Glob", "Bash"]`

### Changes from Previous Format

**Note from official docs:** "In version 2.1.63, the Task tool was renamed to Agent. Existing `Task(...)` references in settings and agent definitions still work as aliases."

This is the only breaking change explicitly documented. The `Task(...)` syntax in `tools` field is now `Agent(...)` — but old `Task(...)` references still work as aliases.

No other breaking changes from previous formats found in official documentation.

### What to Avoid / Obsolete Content

1. `Task(...)` syntax in the `tools` field is now `Agent(...)` — though old syntax still works as alias. Prefer `Agent(...)` for new agent definitions.
2. Do NOT use `bypassPermissions` in `permissionMode` unless the parent session was started with `--dangerously-skip-permissions` — the mode has no effect otherwise.
3. Subagents **cannot spawn other subagents** — the Agent tool restriction in frontmatter only applies to agents running as main thread with `claude --agent`.
4. Do NOT add `Agent(agent_type)` to a subagent's `tools` field expecting it to spawn agents — subagents cannot spawn other subagents, so this restriction has no effect.
5. `context: fork` (skill field) does NOT apply to agent definition files — only to `SKILL.md` files.
6. The `user-invocable` and `disable-model-invocation` fields are **skill fields**, NOT agent fields. Agent delegation is controlled by `description` and global permission settings.
