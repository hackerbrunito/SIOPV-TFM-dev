## TOPIC 1: Hook Script Formats + All Valid Event Types

### Complete Event Type List

Source: Official Claude Code documentation at https://code.claude.com/docs/en/hooks (fetched 2026-03-12)

| Event | Blocking (exit 2)? | When fires |
|-------|-------------------|------------|
| `SessionStart` | No — stderr shown to user only | Session begins or resumes |
| `InstructionsLoaded` | No — exit code ignored | CLAUDE.md or `.claude/rules/*.md` file loaded |
| `UserPromptSubmit` | YES — blocks prompt, erases it | User submits a prompt, before Claude processes |
| `PreToolUse` | YES — blocks the tool call | Before a tool call executes |
| `PermissionRequest` | YES — denies the permission | When a permission dialog appears |
| `PostToolUse` | No — stderr shown to Claude | After a tool call succeeds |
| `PostToolUseFailure` | No — stderr shown to Claude | After a tool call fails |
| `Notification` | No — stderr shown to user only | Claude Code sends a notification |

### Per-Event-Type Details

#### SessionStart
- **When fires:** When a session begins (new) or resumes (--resume, --continue, /resume, /clear, /compact)
- **Matcher values:** `startup` (new session), `resume`, `clear`, `compact`
- **Inputs (stdin JSON):**
  ```json
  {
    "session_id": "abc123",
    "transcript_path": "/Users/.../.claude/projects/.../transcript.jsonl",
    "cwd": "/Users/...",
    "permission_mode": "default",
    "hook_event_name": "SessionStart",
    "source": "startup",
    "model": "claude-sonnet-4-6"
  }
  ```
  Optional fields when using `--agent`: `agent_type`
- **Special env var:** `CLAUDE_ENV_FILE` — write `export VAR=value` lines here to persist env vars to subsequent Bash commands. Only available to SessionStart hooks.
- **Stdout behavior:** Any text printed to stdout is added as context for Claude
- **JSON output option:** `hookSpecificOutput.additionalContext` — string added to Claude's context
- **Blocking:** No. Exit 2 shows stderr to user only.
- **Hook types supported:** Only `type: "command"` (no prompt/agent/http)

#### InstructionsLoaded
- **When fires:** When CLAUDE.md or `.claude/rules/*.md` files are loaded. Fires at session start and when files are lazily loaded (nested CLAUDE.md access, conditional `paths:` frontmatter match, `@include` directives).
- **Matcher:** No matcher support — always fires
- **Inputs (stdin JSON):**
  ```json
  {
    "session_id": "abc123",
    "transcript_path": "...",
    "cwd": "/Users/my-project",
    "permission_mode": "default",
    "hook_event_name": "InstructionsLoaded",
    "file_path": "/Users/my-project/CLAUDE.md",
    "memory_type": "Project",
    "load_reason": "session_start"
  }
  ```
  Additional fields: `globs` (for `path_glob_match` loads), `trigger_file_path` (for lazy loads), `parent_file_path` (for `include` loads)
- **load_reason values:** `"session_start"`, `"nested_traversal"`, `"path_glob_match"`, `"include"`
- **memory_type values:** `"User"`, `"Project"`, `"Local"`, `"Managed"`
- **Blocking:** No. Exit code is ignored.
- **Decision control:** None. Use for audit logging / compliance tracking only.

#### UserPromptSubmit
- **When fires:** When user submits a prompt, before Claude processes it
- **Matcher:** No matcher support
- **Inputs (stdin JSON):**
  ```json
  {
    "session_id": "abc123",
    "transcript_path": "...",
    "cwd": "/Users/...",
    "permission_mode": "default",
    "hook_event_name": "UserPromptSubmit",
    "prompt": "Write a function to calculate the factorial of a number"
  }
  ```
- **Stdout behavior on exit 0:** Non-JSON text printed to stdout is added as context (shown in transcript). JSON `hookSpecificOutput.additionalContext` added more discretely.
- **To block:** `{"decision": "block", "reason": "..."}` — reason shown to user, prompt erased
- **Blocking:** YES via exit 2 OR via JSON `decision: "block"`

#### PreToolUse
- **When fires:** After Claude creates tool parameters, before tool call executes
- **Matcher:** Tool name (regex). Examples: `Bash`, `Edit|Write`, `mcp__.*`
- **Inputs (stdin JSON):**
  ```json
  {
    "session_id": "abc123",
    "transcript_path": "...",
    "cwd": "/Users/...",
    "permission_mode": "default",
    "hook_event_name": "PreToolUse",
    "tool_name": "Bash",
    "tool_use_id": "toolu_01ABC...",
    "tool_input": {
      "command": "npm test"
    }
  }
  ```
  `tool_input` schema varies by tool (Bash: `command`, `description`, `timeout`, `run_in_background`; Write: `file_path`, `content`; Edit: `file_path`, `old_string`, `new_string`, `replace_all`; Read: `file_path`, `offset`, `limit`; Glob: `pattern`, `path`; Grep: `pattern`, `path`, `glob`, `output_mode`, `-i`, `multiline`; WebFetch: `url`, `prompt`; WebSearch: `query`, `allowed_domains`, `blocked_domains`; Agent: `prompt`, `description`, `subagent_type`, `model`)
- **Decision control:** Uses `hookSpecificOutput` (NOT top-level `decision`)
  ```json
  {
    "hookSpecificOutput": {
      "hookEventName": "PreToolUse",
      "permissionDecision": "allow|deny|ask",
      "permissionDecisionReason": "...",
      "updatedInput": {"field": "new_value"},
      "additionalContext": "..."
    }
  }
  ```
  - `"allow"`: bypass permission system
  - `"deny"`: prevent tool call (reason shown to Claude)
  - `"ask"`: prompt user to confirm
- **DEPRECATION NOTE:** PreToolUse previously used top-level `decision`/`reason`. Now deprecated. Use `hookSpecificOutput.permissionDecision` and `hookSpecificOutput.permissionDecisionReason`. Old values `"approve"` → `"allow"`, `"block"` → `"deny"` still work.
- **Blocking:** YES via exit 2 OR via `hookSpecificOutput.permissionDecision: "deny"`

#### PermissionRequest
- **When fires:** When a permission dialog is about to be shown to user
- **Difference from PreToolUse:** PreToolUse fires before all tool executions regardless of permission status. PermissionRequest fires only when a permission dialog would appear.
- **Matcher:** Tool name, same values as PreToolUse
- **Inputs (stdin JSON):**
  ```json
  {
    "session_id": "abc123",
    "transcript_path": "...",
    "cwd": "/Users/...",
    "permission_mode": "default",
    "hook_event_name": "PermissionRequest",
    "tool_name": "Bash",
    "tool_input": {"command": "rm -rf node_modules"},
    "permission_suggestions": [
      {"type": "toolAlwaysAllow", "tool": "Bash"}
    ]
  }
  ```
  Note: no `tool_use_id` unlike PreToolUse
- **Decision control:**
  ```json
  {
    "hookSpecificOutput": {
      "hookEventName": "PermissionRequest",
      "decision": {
        "behavior": "allow|deny",
        "updatedInput": {"command": "npm run lint"},
        "updatedPermissions": [...],
        "message": "reason for deny",
        "interrupt": true
      }
    }
  }
  ```
- **Blocking:** YES via exit 2 (denies permission)

#### PostToolUse
- **When fires:** Immediately after a tool completes successfully
- **Matcher:** Tool name, same values as PreToolUse
- **Inputs (stdin JSON):**
  ```json
  {
    "session_id": "abc123",
    "transcript_path": "...",
    "cwd": "...",
    "permission_mode": "default",
    "hook_event_name": "PostToolUse",
    "tool_name": "Write",
    "tool_use_id": "toolu_01ABC...",
    "tool_input": {"file_path": "/path/to/file.txt", "content": "..."},
    "tool_response": {"filePath": "/path/to/file.txt", "success": true}
  }
  ```
- **Decision control:** Top-level `decision: "block"` + `reason` (feeds back to Claude). Also supports `additionalContext` and `updatedMCPToolOutput` (for MCP tools, replaces tool output).
- **Blocking:** No (tool already ran). Exit 2 shows stderr to Claude.

#### PostToolUseFailure
- **When fires:** When a tool execution fails
- **Matcher:** Tool name
- **Inputs:** Same as PostToolUse plus:
  - `error` (string): what went wrong
  - `is_interrupt` (boolean, optional): whether caused by user interruption
- **Decision control:** Only `additionalContext` via `hookSpecificOutput`
- **Blocking:** No. Exit 2 shows stderr to Claude.

#### Notification
- **When fires:** When Claude Code sends notifications
- **Matcher values:** `permission_prompt`, `idle_prompt`, `auth_success`, `elicitation_dialog`
- **Inputs:**
  ```json
  {
    "...",
    "hook_event_name": "Notification",
    "message": "Claude needs your permission to use Bash",
    "title": "Permission needed",
    "notification_type": "permission_prompt"
  }
  ```
- **Decision control:** Cannot block notifications. Can return `additionalContext`.
- **Blocking:** No. Exit 2 shows stderr to user only.

#### SubagentStart
- **When fires:** When a Claude Code subagent is spawned via the Agent tool
- **Matcher:** Agent type name (`Bash`, `Explore`, `Plan`, or custom agent names from `.claude/agents/`)
- **Inputs:** Common fields + `agent_id`, `agent_type`
- **Decision control:** Cannot block. Can return `additionalContext` to inject into subagent context.
- **Blocking:** No.

#### SubagentStop
- **When fires:** When a Claude Code subagent finishes responding
- **Matcher:** Agent type (same values as SubagentStart)
- **Decision control:** Top-level `decision: "block"` prevents subagent from stopping
- **Blocking:** YES via exit 2

#### Stop
- **When fires:** When Claude finishes responding
- **Matcher:** No matcher support
- **Decision control:** Top-level `decision: "block"` + `reason` prevents stopping, continues conversation with reason as next instruction
- **Prompt/agent hooks:** Supported. LLM returns `{"ok": true}` to allow stop, `{"ok": false, "reason": "..."}` to continue.
- **Blocking:** YES via exit 2

#### TeammateIdle
- **When fires:** When an agent team teammate is about to go idle
- **Matcher:** No matcher support
- **Decision control:** Exit code 2 blocks idle (teammate continues working). JSON `{"continue": false, "stopReason": "..."}` stops the teammate entirely.
- **Blocking:** YES via exit 2

#### TaskCompleted
- **When fires:** When a task is being marked as completed
- **Matcher:** No matcher support
- **Decision control:** Exit code 2 prevents task from being marked complete. JSON `{"continue": false}` also stops teammate entirely.
- **Blocking:** YES via exit 2

#### ConfigChange
- **When fires:** When a configuration file changes during a session
- **Matcher values:** `user_settings`, `project_settings`, `local_settings`, `policy_settings`, `skills`
- **Note:** Only `type: "command"` hooks supported
- **Decision control:** Top-level `decision: "block"` prevents config change (EXCEPT for `policy_settings` source — cannot block managed settings)
- **Blocking:** YES via exit 2 (except policy_settings)

#### WorktreeCreate
- **When fires:** When a worktree is being created via `--worktree` or `isolation: "worktree"`
- **Matcher:** No matcher support
- **Note:** Replaces default git behavior. Hook prints absolute path to created worktree on stdout.
- **Decision control:** Non-zero exit code fails worktree creation. Stdout path replaces the default worktree.
- **Blocking:** YES (any non-zero exit code fails creation)

#### WorktreeRemove
- **When fires:** When a worktree is being removed (session exit or subagent completion)
- **Matcher:** No matcher support
- **Decision control:** None. Failures logged in debug mode only.
- **Blocking:** No.

#### PreCompact
- **When fires:** Before context compaction
- **Matcher values:** `manual`, `auto`
- **Note:** Only `type: "command"` hooks supported
- **Decision control:** None (cannot block compaction)
- **Blocking:** No. Exit 2 shows stderr to user only.

#### SessionEnd
- **When fires:** When a session terminates
- **Matcher values:** `clear`, `logout`, `prompt_input_exit`, `bypass_permissions_disabled`, `other`
- **Note:** Only `type: "command"` hooks supported
- **Decision control:** None
- **Blocking:** No. Exit 2 shows stderr to user only.

### Hook Script Format

- **Script type:** Any shell script (bash most commonly shown in docs)
- **Shebang:** Required for shell scripts (e.g., `#!/bin/bash`)
- **Executable bit:** Not explicitly required in docs — command is run as shell command string (e.g., `"command": "bash .claude/hooks/script.sh"` or `"command": ".claude/hooks/script.sh"`)
- **Input:** JSON received via stdin for command hooks. For HTTP hooks, arrives as POST request body.
- **Output (exit codes):**
  - Exit 0: success. Claude Code parses stdout for JSON output fields. Non-JSON stdout behavior depends on event (UserPromptSubmit and SessionStart add it as context; most others only visible in verbose mode via Ctrl+O).
  - Exit 2: blocking error. Claude Code ignores stdout/JSON. stderr is fed back to Claude (or shown to user for non-blocking events).
  - Any other non-zero: non-blocking error. stderr shown in verbose mode.
- **Output (JSON on stdout):** Hook stdout must contain ONLY the JSON object on exit 0. Shell profile startup text can break JSON parsing.
- **Universal JSON output fields:**
  - `continue` (default: true): if false, Claude stops processing entirely. Takes precedence over event-specific decisions.
  - `stopReason`: message shown to user when `continue` is false (not shown to Claude)
  - `suppressOutput` (default: false): if true, hides stdout from verbose mode
  - `systemMessage`: warning message shown to user

- **Four hook handler types:**
  1. `type: "command"` — shell command, stdin/stdout/exit codes
  2. `type: "http"` — HTTP POST to URL, same JSON output schema
  3. `type: "prompt"` — LLM single-turn evaluation (returns `{"ok": true/false, "reason": "..."}`)
  4. `type: "agent"` — spawns subagent with tools (Read, Grep, Glob) for verification

- **Events supporting ALL four types (command, http, prompt, agent):** PermissionRequest, PostToolUse, PostToolUseFailure, PreToolUse, Stop, SubagentStop, TaskCompleted, UserPromptSubmit
- **Events supporting ONLY `type: "command"`:** ConfigChange, Notification, PreCompact, SessionEnd, SessionStart, SubagentStart, TeammateIdle, WorktreeCreate, WorktreeRemove

### Registration Mechanism

Hooks are registered in JSON settings files under the top-level `"hooks"` key. Three nesting levels:
1. Hook event name (e.g., `"PreToolUse"`)
2. Array of matcher group objects, each with optional `"matcher"` (regex string) and `"hooks"` array
3. Hook handler objects with `"type"`, `"command"` (or `"url"`/`"prompt"`), and optional fields

```json
{
  "hooks": {
    "PreToolUse": [
      {
        "matcher": "Bash",
        "hooks": [
          {
            "type": "command",
            "command": ".claude/hooks/block-rm.sh",
            "timeout": 30
          }
        ]
      }
    ]
  }
}
```

**Hook locations (scope table):**

| Location | Scope | Shareable |
|----------|-------|-----------|
| `~/.claude/settings.json` | All projects | No |
| `.claude/settings.json` | Single project | Yes (committed) |
| `.claude/settings.local.json` | Single project | No (gitignored) |
| Managed policy settings | Organization-wide | Admin-controlled |
| Plugin `hooks/hooks.json` | When plugin enabled | Yes |
| Skill/agent frontmatter | While component active | Yes |

**Alternative hook definition in skill/agent YAML frontmatter:**
```yaml
---
name: secure-operations
hooks:
  PreToolUse:
    - matcher: "Bash"
      hooks:
        - type: command
          command: "./scripts/security-check.sh"
---
```

**Interactive management:** Type `/hooks` in Claude Code to open hooks manager (only supports adding command hooks; HTTP hooks must be added by editing JSON directly).

**Hook snapshot behavior:** Claude Code captures hooks at startup. External modifications don't take effect mid-session — requires review in `/hooks` menu before changes apply.

### Hook Handler Common Fields

| Field | Required | Description |
|-------|----------|-------------|
| `type` | yes | `"command"`, `"http"`, `"prompt"`, or `"agent"` |
| `timeout` | no | Seconds before cancel. Defaults: 600 (command), 30 (prompt), 60 (agent) |
| `statusMessage` | no | Custom spinner message while hook runs |
| `once` | no | If true, runs only once per session (skills only) |

**Command hook additional fields:**
- `command` (required): shell command string
- `async` (optional): if true, runs in background without blocking

**HTTP hook additional fields:**
- `url` (required): POST target URL
- `headers` (optional): key-value pairs, support `$VAR_NAME` interpolation
- `allowedEnvVars` (optional): env vars allowed to be interpolated into headers

**Prompt/agent hook additional fields:**
- `prompt` (required): text to send to model. Use `$ARGUMENTS` as placeholder for hook input JSON
- `model` (optional): model to use, defaults to fast model

### Environment Variables Available to Hook Scripts

- `CLAUDE_PROJECT_DIR`: project root directory (use in `"command"` field to reference scripts)
- `${CLAUDE_PLUGIN_ROOT}`: plugin root directory (for plugin-bundled scripts)
- `CLAUDE_ENV_FILE`: (SessionStart only) path to write `export VAR=value` lines for session-wide env vars
- `$CLAUDE_CODE_REMOTE`: set to `"true"` in remote web environments, unset in local CLI
- All env vars from `env` section of settings.json
- Claude Code runs hooks with the same environment variables as the session

### Minimal Working Examples

#### Blocking example (PreToolUse — blocks dangerous rm -rf):
```bash
#!/bin/bash
# .claude/hooks/block-rm.sh
COMMAND=$(jq -r '.tool_input.command')

if echo "$COMMAND" | grep -q 'rm -rf'; then
  jq -n '{
    hookSpecificOutput: {
      hookEventName: "PreToolUse",
      permissionDecision: "deny",
      permissionDecisionReason: "Destructive command blocked by hook"
    }
  }'
else
  exit 0
fi
```

Registered in settings.json:
```json
{
  "hooks": {
    "PreToolUse": [
      {
        "matcher": "Bash",
        "hooks": [{ "type": "command", "command": ".claude/hooks/block-rm.sh" }]
      }
    ]
  }
}
```

#### Non-blocking example (SessionStart — load project context):
```bash
#!/bin/bash
# .claude/hooks/load-context.sh
cd "$CLAUDE_PROJECT_DIR" || exit 1

if [ -f "package.json" ]; then
  echo "Node.js project detected"
  echo "export PROJECT_TYPE=nodejs" >> "$CLAUDE_ENV_FILE"
fi

exit 0
```

Registered:
```json
{
  "hooks": {
    "SessionStart": [
      {
        "matcher": "startup",
        "hooks": [{ "type": "command", "command": "\"$CLAUDE_PROJECT_DIR\"/.claude/hooks/load-context.sh" }]
      }
    ]
  }
}
```

#### Non-blocking example (PostToolUse — run linter after file write):
```bash
#!/bin/bash
input=$(cat)
file_path=$(echo "$input" | jq -r '.tool_input.file_path')

if [[ "$file_path" == *.js ]] || [[ "$file_path" == *.ts ]]; then
  npx eslint "$file_path" 2>&1 || true
fi
exit 0
```

### Hook Registration Syntax

**CONFLICT DETECTED:**

The official hooks reference (https://code.claude.com/docs/en/hooks) uses camelCase event names:
```json
{
  "hooks": {
    "PreToolUse": [...],
    "PostToolUse": [...],
    "SessionStart": [...]
  }
```

### Project-level vs Global settings.json

| Aspect | Global `~/.claude/settings.json` | Project `.claude/settings.json` | Local `.claude/settings.local.json` |
|--------|----------------------------------|----------------------------------|--------------------------------------|
| Scope | All projects | Single repository | Single project |
| Shareable | No (personal only) | Yes (commit to git) | No (gitignored) |
| Use cases | Personal preferences, API keys | Team standards, project MCP servers | Personal project overrides |
| Precedence | Lowest | Overrides global | Overrides project |

**Fourth location:** Managed policy settings (organization-wide, admin-controlled, highest precedence)

**`~/.claude.json`** (separate from settings.json): Stores global state — theme preferences, OAuth tokens, MCP server details. Not for user configuration.

**`.mcp.json`**: Project-specific MCP server configurations (separate file from settings.json).

**Array settings (e.g., `permissions.allow`, `hooks`):** MERGE across scopes (user + project + managed combined), not override.

**Precedence for scalar values:** managed policy > local > project > global (user)
