# Fix Report: structlog format_exc_info Deprecation

**Date:** 2026-03-15
**File:** src/siopv/infrastructure/logging/setup.py
**Issue:** Known #9 — `structlog.processors.format_exc_info` deprecation warning

## Root Cause Analysis

The issue had two layers:

1. **Immediate:** `structlog.processors.format_exc_info` (a deprecated global alias) was used in
   `shared_processors` instead of the class-based `structlog.processors.ExceptionRenderer()`.

2. **Deeper:** `ExceptionRenderer()` in `shared_processors` conflicts with `ConsoleRenderer` in dev mode.
   `ConsoleRenderer` handles `exc_info` natively and emits a `UserWarning` when it detects a pre-rendered
   `exception` key already in the event dict (placed there by `ExceptionRenderer()`).

## Fix Applied

Removed `ExceptionRenderer()` from `shared_processors` entirely (it is only needed for JSON output,
not for `ConsoleRenderer` which handles exceptions natively).

Added `ExceptionRenderer()` exclusively to the JSON formatter's processor chain, placed before
`JSONRenderer()` so that `exc_info` tuples are converted to strings before JSON serialization.

### Before
```python
shared_processors = [
    ...
    structlog.processors.format_exc_info,   # deprecated + conflicts with ConsoleRenderer
    ...
]
formatter = structlog.stdlib.ProcessorFormatter(
    foreign_pre_chain=shared_processors,
    processors=[remove_meta, renderer],
)
```

### After
```python
shared_processors = [
    ...
    # ExceptionRenderer removed — ConsoleRenderer handles exc_info natively
    ...
]

if json_format:
    formatter_processors = [remove_meta, ExceptionRenderer(), JSONRenderer()]
else:
    formatter_processors = [remove_meta, ConsoleRenderer()]  # handles exc_info itself

formatter = structlog.stdlib.ProcessorFormatter(
    foreign_pre_chain=shared_processors,
    processors=formatter_processors,
)
```
