## 1. PT006: Parametrize Arguments Type (Tuple vs List)

### Current Best Practice (2026)
**Use TUPLE format for multiple parameters:**

```python
# ✅ CORRECT (Tuple - PT006 compliant)
@pytest.mark.parametrize(("name1", "name2"), [
    ("value1", "value2"),
    ("value3", "value4"),
])
def test_function(name1, name2):
    ...

# ✅ CORRECT (String for single parameter)
@pytest.mark.parametrize("name", ["value1", "value2"])
def test_function(name):
    ...

# ❌ WRONG (List - violates PT006)
@pytest.mark.parametrize(["name1", "name2"], [...])

# ⚠️  ACCEPTABLE (CSV - depends on config)
@pytest.mark.parametrize("name1,name2", [...])
```

### Key Rules
- **Single parameter:** Use string (never tuple/list)
- **Multiple parameters:** Use tuple (default enforced by PT006)
- **Values:** Always provide list of tuples for test values
- **Linter config:** Ruff/flake8-pytest-style enforces tuple by default

**Sources:**
- [PT006 Rule - Ruff](https://docs.astral.sh/ruff/rules/pytest-parametrize-names-wrong-type/)
- [flake8-pytest-style PT006](https://github.com/m-burst/flake8-pytest-style/blob/master/docs/rules/PT006.md)

## 2. PT011: pytest.raises Match Parameter

### Current Best Practice (2026)
**ALWAYS use `match` parameter with pytest.raises:**

```python
# ✅ CORRECT (PT011 compliant)
with pytest.raises(ValueError, match=r"expected message"):
    myfunc()

# ✅ CORRECT (Regex pattern for flexible matching)
with pytest.raises(ValueError, match=r"Invalid input: \d+"):
    process_data(-1)

# ❌ WRONG (Too broad - violates PT011)
with pytest.raises(ValueError):
    myfunc()  # May catch unrelated ValueError!
```

### Key Rules
- **match parameter is MANDATORY** (prevents catching unrelated exceptions)
- **Uses re.search()** under the hood (regex matching)
- **Be precise with patterns** to avoid false positives
- **Configurable exceptions:** `lint.flake8-pytest-style.raises-require-match-for`

### Why This Matters
`pytest.raises(Error)` catches ANY Error, including errors from unrelated code executed during the test, leading to false positives.

**Sources:**
- [PT011 Rule - Ruff](https://docs.astral.sh/ruff/rules/pytest-raises-too-broad/)
- [pytest.raises match parameter - mathspp](https://mathspp.com/blog/til/pytest.raises-parameter-match)

## 3. ARG002: Unused Fixture Arguments

### Current Best Practice (2026)
**When fixtures are used for side effects (not return values), use `@pytest.mark.usefixtures`:**

```python
# ✅ CORRECT (No ARG002 violation)
@pytest.mark.usefixtures("_patch_api", "_setup_db")
def test_function():
    # Fixtures activate automatically, no unused args
    ...

# ✅ ACCEPTABLE (Underscore prefix for implicit fixtures)
def test_function(_setup_db):
    # Underscore prefix suppresses ARG002 warning
    ...

# ❌ WRONG (Triggers ARG002)
def test_function(setup_db):  # setup_db unused in body
    assert some_condition()
```

### Key Rules
- **Prefer `@pytest.mark.usefixtures`** for side-effect fixtures
- **Underscore prefix** suppresses ARG002 for fixture names
- **Multiple fixtures:** `@pytest.mark.usefixtures("fix1", "fix2", "fix3")`
- **Limitation:** Cannot use on other fixtures (only on test functions)

### When Fixtures ARE Used
If the fixture return value is actually used, no special handling needed:

```python
def test_function(database_session):
    result = database_session.query(...).all()  # ✅ Used, no ARG002
    assert result
```

## 4. Conftest.py Patterns & Test Organization

### Current Best Practice (2026)
**Directory-based fixture scoping with modular conftest.py files:**

```
tests/
├── conftest.py              # Global fixtures (all tests)
├── unit/
│   ├── conftest.py          # Unit-test specific fixtures
│   └── test_models.py
├── integration/
│   ├── conftest.py          # Integration-test specific fixtures
│   └── test_api.py
└── e2e/
    ├── conftest.py          # E2E-test specific fixtures
    └── test_workflows.py
```

### Key Organizational Patterns (2026)
1. **File layout as base layer** (group by domain/layer)
2. **Markers for cross-cutting concerns** (slow, integration, requires_api)
3. **Classes for related tests in same file** (when file too large)
4. **Multiple conftest files for modularity** (avoid monolithic conftest.py)

### Fixture Scoping Rules
- **Nearest conftest wins:** Fixtures from subfolder conftest.py override parent
- **Automatic discovery:** No imports needed, pytest finds fixtures up the tree
- **Reusability:** Domain-specific fixtures only affect tests in their subtree

### Modern Patterns (2026)
```python
# conftest.py - Domain-specific setup
@pytest.fixture(scope="session")
def api_client():
    """Shared API client for integration tests."""
    return APIClient(base_url="https://api.example.com")

@pytest.fixture(scope="function")
def clean_database(database_session):
    """Ensure clean DB state per test."""
    yield
    database_session.rollback()
```

## 5. Test Naming & Organization Best Practices

### File Naming
- **Prefix:** `test_*.py` or `*_test.py` (pytest auto-discovery)
- **Descriptive:** `test_user_authentication.py` (not `test1.py`)

### Function Naming
```python
# ✅ CORRECT (Descriptive, follows convention)
def test_user_login_with_valid_credentials():
    ...

def test_user_login_raises_error_for_invalid_password():
    ...

# ❌ WRONG (Vague, non-descriptive)
def test_login():
    ...

def test_error():
    ...
```

### Class-Based Organization
```python
class TestUserAuthentication:
    """Group related authentication tests."""

    def test_login_success(self):
        ...

    def test_login_failure_invalid_password(self):
        ...

    def test_logout_clears_session(self):
        ...
```

## 6. Pytest 8 Breaking Changes (from Pytest 7)

### Collection Phase Changes
1. **Alphabetical collection:** Files and directories now collected alphabetically jointly
2. **Overlapping arguments:** `pytest a/ a/b` treated as `pytest a` (prefix only)
3. **Package handling:** Improved collection of Python packages

### API & Configuration Changes
1. **Python 3.7 dropped:** Minimum version now Python 3.8+
2. **junit_family default:** Changed from `xunit1` to `xunit2`
   - **Migration:** Add `junit_family=xunit1` to config if needed
3. **TerminalReporter.writer removed:** Use public TerminalReporter functions instead
4. **--result-log removed:** Use `pytest-reportlog` plugin instead

### No Deprecation Period
Breaking changes to collection phase had **no deprecation period** due to fixing fundamental deficiencies. Projects upgrading from pytest 7 should test thoroughly.
